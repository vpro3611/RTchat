Design: Reply Data Flow

How a message becomes a reply

1. Sending a Reply
  1. Client sends { content: "My reply", parentMessageId: "UUID-A" }
  2. UseCase finds Message A, takes a snippet of its content.
  3. Repository saves "My reply" as a new message in messages.
  4. Repository saves link in message_replies with the snippet.
2. Reading Messages

SELECT m.*, r.parent_message_id, r.parent_content_snippet
FROM messages m
LEFT JOIN message_replies r ON m.id = r.message_id
WHERE m.conversation_id = ...

We get the reply context in a single query!

Does this flow look right to you?

  • Storing a snippet in the link table avoids expensive recursive joins or fetching full parent messages every time.
  • It handles "deleted" parents gracefully (the snippet remains).
